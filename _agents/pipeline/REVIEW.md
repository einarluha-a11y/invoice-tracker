ВЕРДИКТ: CHANGES_NEEDED

ОЦЕНКА: Архитектурное решение логично структурировано с трёхуровневой моделью доступа и чёткой иерархией данных (system → accounts → companies). Предложенная структура Firestore соответствует принципам многоуровневой архитектуры[4] с разделением функций и независимой масштабируемостью каждого уровня. Модульный подход позволит разработчикам работать над компонентами параллельно.

ЗАМЕЧАНИЯ:

1. **Отсутствие обработки edge case: пользователь удалён из аккаунта во время сессии** — нужна проверка прав при каждом запросе, не только при входе. Риск: пользователь сохранит доступ к данным через кэшированный токен.

2. **Миграция данных требует атомарности** — текущие Global Technics и Ideacom должны мигрировать в `accounts/{accountId}/companies/` одной транзакцией. Если процесс прервётся, данные окажутся в несогласованном состоянии. Нужен rollback-план.

3. **imap_daemon.cjs будет загружать все компании всех аккаунтов** — это создаст O(n) запросов при каждом цикле. Предложить кэширование списка компаний с TTL или использовать Firestore collection group queries.

4. **Security Rules не описаны** — нужны явные правила для трёх ролей. Риск: неправильные правила откроют доступ между аккаунтами или позволят user редактировать поле role.

5. **Поле `processed_email_uids` в структуре** — не ясна его роль. Если это кэш обработанных писем, нужна стратегия инвалидации, иначе новые письма могут быть пропущены.

6. **Login.tsx: поле ввода названия аккаунта** — уязвимость: пользователь может перебором узнать названия всех аккаунтов. Предложить либо скрыть список, либо требовать точное совпадение с проверкой прав после Google входа.

7. **api_server.cjs auth middleware** — не указано, как передавать accountId в контексте запроса. Нужна стратегия: в URL, в заголовке или в JWT payload.

8. **Downtime-free миграция невозможна** — старая структура (без accounts/) и новая несовместимы. Нужно либо двойное написание (write to both), либо плановый downtime с откатом.
